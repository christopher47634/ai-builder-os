# Demo Script & Interview Guide — AI Builder OS

## 30 秒版本（电梯演讲）

"AI Builder OS 是一个面向 AI 产品工程师的综合工作台。它整合了 100 个 AI 项目模板、100 个客户接单案例、200 个工作流方案和 600 条评测用例，覆盖从项目发现到商业化交付的全流程。技术栈是 Next.js 16 + TypeScript，1,000 条数据本地加载，零外部依赖。"

## 1 分钟版本

"AI Builder OS 解决一个真实问题：AI 工程师想接单做项目，需要在 4-5 个不同工具之间切换——查项目库、分析客户、找工作方案、评估系统质量。我把这 4 步整合到一个工作台里。

核心模块：100 个 AI 项目有七维度评分，支持筛选和一键生成简历；100 个客户案例构建了规则引擎，输入客户原话就能分析需求、评估风险、生成报价；200 个工作流模板覆盖 20 个方向，每个有完整工具链和 Prompt；600 条评测用例覆盖 RAG、安全、幻觉等 6 个维度。

技术栈 Next.js 16 + TypeScript + Tailwind CSS，1,000 条结构化数据本地加载，零外部依赖。"

## 3 分钟版本

**问题（30秒）**：
AI 应用领域项目分散、工具碎片化。一个 AI 工程师想找到值得做的项目、接到合适的单子、用对工作流模板、评估系统质量，需要在 4-5 个不同工具和资料库之间切换。

**方案（30秒）**：
AI Builder OS 把这 4 个环节整合到一个工作台里。100 个项目有七维度评分，100 个客户案例构建了分析引擎，200 个工作流有完整方案，600 条评测覆盖 6 个维度。

**技术实现（1分钟）**：
Next.js 16 App Router + TypeScript strict mode + Tailwind CSS。数据层用本地 JSON，1,000 条结构化数据从 4 个异构数据源提取——Markdown 文档、JSON 文件、TypeScript CLI、Next.js 应用。客户分析用纯前端规则引擎，不依赖外部 LLM。评测预留 OpenAI-compatible Provider 接口，支持 Mock 和真实 LLM 双模式。

**成果（30秒）**：
7 个页面全部编译通过，npm run build 零错误，TypeScript 零 any 类型。深色玻璃拟态主题，响应式布局，移动端适配。这个项目展示了数据工程、产品设计、全栈开发、AI 评测和商业化交付 5 个维度的能力。

## 面试官追问 20 个问题

### 项目动机

**Q1: 为什么做这个项目？**
"AI 工程师想接单做项目，需要查 4-5 个不同工具。我把这 4 步整合到一个界面里，解决真实的效率问题。"

**Q2: 这个项目的目标用户是谁？**
"三类人：AI 产品经理想了解可做的项目方向，AI 工程师想接单找项目，自由职业者想用工作流模板提效。"

**Q3: 你是怎么确定这 7 个页面的？**
"从用户旅程出发：先发现项目（项目库），再分析客户（接单分析），再找方案（工作流），再评估质量（评测），最后包装自己（作品集）。Dashboard 是总览，Status 是验收。"

### 数据来源

**Q4: 100 个 AI 项目的数据从哪来？**
"从结构化 Markdown 文档提取，每个项目有 50+ 字段，包括七维度评分、痛点分析、核心功能、技术栈、MVP 范围。用 Python 脚本解析 Markdown 列表和嵌套字段。"

**Q5: 100 个客户案例是真实的吗？**
"是模拟案例，但基于真实客户画像设计。覆盖 12 种客户类型（小公司老板、企业管理层、自由职业者等），每个案例有完整的沟通技巧、成交话术和风险提示。"

**Q6: 600 条评测用例怎么设计的？**
"6 类 × 100 条：RAG 检索准确性、多轮对话一致性、工具调用正确性、幻觉检测、安全边界、恶意输入防御。每条有标准答案、评分标准和通过条件。"

### 技术架构

**Q7: 为什么选 Next.js 而不是 Vite + React？**
"Next.js 的静态生成可以让 7 个页面预渲染为 HTML，部署时零服务器依赖。App Router 的文件系统路由天然适合多页面应用。"

**Q8: 数据为什么用本地 JSON 而不是数据库？**
"作品集项目的核心是展示能力，不是运维。本地 JSON 零配置、零依赖、零成本，npm install 就能跑。未来需要时可以无缝切换到 SQLite 或 PostgreSQL。"

**Q9: 客户分析引擎怎么实现的？**
"纯前端规则引擎。解析客户原话中的关键词（如'客服''自动回复''多少钱'），匹配预定义的模式库，输出客户类型、需求层次、风险点和报价建议。不依赖外部 LLM。"

**Q10: TypeScript 严格模式带来了什么好处？**
"零 any 类型意味着所有数据结构都有明确定义。1,000 条 JSON 数据的每个字段都有对应的 interface，页面代码不会出现运行时类型错误。"

### 商业价值

**Q11: 这个项目怎么变现？**
"三种方式：第一，作为作品集帮助找工作/接单；第二，接入真实 LLM 后可以作为 SaaS 产品；第三，工作流模板和项目库本身可以作为付费内容。"

**Q12: 接真实 LLM 需要改多少？**
"客户分析模块需要把规则引擎替换为 LLM 调用，接口已经预留（OpenAI-compatible Provider）。评测模块的 Provider 接口也已定义。预计 1-2 周可以接入。"

**Q13: 这个项目和市面上的产品有什么不同？**
"Dify、FastGPT 是部署平台，这个是决策工具。它帮你在做项目之前做判断：这个项目值不值得做？这个客户值不值得接？用什么方案最划算？"

### 不足与改进

**Q14: 这个项目有什么不足？**
"三个主要不足：第一，客户分析是规则引擎，不是真正的 AI，复杂场景判断不够准；第二，评测是 Mock 模式，没有真正跑 LLM 推理；第三，没有数据持久化，刷新页面就重置。"

**Q15: 如果再做一次，你会改什么？**
"第一，先做数据持久化（localStorage），让用户分析结果可以保存；第二，接入真实 LLM 做客户分析，提升判断准确性；第三，加 a11y 支持，让项目更专业。"

**Q16: 下一步打算怎么升级？**
"短期：接入 OpenAI-compatible API 做真实客户分析。中期：加数据持久化和 PDF 导出。长期：做成 SaaS 产品，支持团队协作。"

### 个人能力

**Q17: 这个项目展示了你什么能力？**
"五个维度：数据工程（4 异构数据源 → 1,000 条统一数据）、产品设计（7 页面信息架构）、全栈开发（Next.js + TypeScript）、AI 评测（600 条基准）、商业化（客户分析 + 报价）。"

**Q18: 你在这个项目中遇到的最大挑战是什么？**
"数据提取。100 个 AI 项目的 Markdown 结构复杂——列表嵌套列表、代码块、表格混合。写 Python 提取脚本处理了 20+ 种字段格式，包括七维度评分矩阵。"

**Q19: 你花了多少时间做这个项目？**
"从数据整理到产品完成约 2 天。其中数据提取和清洗占 40%，页面开发占 40%，样式和调试占 20%。"

**Q20: 你会推荐别人做类似的项目吗？**
"会，但建议先想清楚目标用户和核心问题。很多人一上来就堆功能，结果做了一堆没人用的页面。先解决一个真实问题，再扩展。"
